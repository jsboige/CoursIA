### C401-L2 : CSV multi-lignes RFC4180 ≠ awk -F, comptage naïf = faux positif (NEW lesson)

`awk -F,` ne respecte **PAS** la spécification RFC4180 (champs multi-lignes entre double-quotes). Quand un CSV contient des cellules avec newlines (markdown pédagogique avec sections, code multi-lignes), **awk voit chaque newline interne comme record separator** = comptage drastiquement erroné (8544 lignes awk vs **567 records Python `csv.DictReader`** = facteur 15×).

**Pattern reaffirmable c.402+** : pour tous les audits CSV dans les sweeps §H.4 cross-owner, **TOUJOURS utiliser Python `csv.DictReader`** (RFC4180-compliant) au lieu de `awk -F,` (naïf line-based). Tests ré-effectués :
- `awk -F,` : **8544 lignes + 758 text_fr populated + 135 text_en populated** (DRASTIQUEMENT FAUX)
- Python `csv.DictReader` : **567 records + 567 text_fr populated + 0 text_en populated** (CORRECT, match body PR)

**Sans Python csv parser, le sweep §H.4 aurait flaggé à tort** : (a) text_en populated = violation T3 gated = advisory bloquant FAUX POSITIF, (b) total records = 8544 vs body annonce 567 = mismatch FAUX POSITIF. Les deux auraient été des advisory **STRUCTURAL_ONLY non-bloquants MAIS parfaitement évitables** par le bon outil.

**Anti-pattern** : `awk -F,` sur CSV avec champs multi-lignes = sweep G.1 erroné. **Pattern** : Python `csv.DictReader` pour audit RFC4180 = sweep G.1 correct.
