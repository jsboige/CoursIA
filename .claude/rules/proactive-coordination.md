# Coordination proactive — 1 PR/wakeup + main track + side-track async

S'applique a **tous les workers du cluster CoursIA** (po-2023/2024/2025/2026) et au **coordinateur ai-01**. Source : mandat user 2026-05-23. Backlog sources + mapping machines + reporting + anti-patterns : [docs/proactive-coordination-detail.md](../../docs/reference/proactive-coordination-detail.md).

## Regles HARD

1. **≥1 PR entre 2 wakeups = PLANCHER, jamais plafond (mandat user 2026-07-01 : « fais en sorte que tous les workers travaillent abondamment entre 2 sessions »).** Une PR livrée ne **clôt pas** la session : re-pioche **immédiatement** dans le pool et **enchaîne autant de PRs que la fenêtre le permet** (débit nominal ~2 PR/h, cf deep-queue). S'arrêter après 1 PR alors qu'il reste du temps de session **et** 60+ issues ouvertes = **sous-régime**, pas « cycle terminé ». Pas d'attente passive, pas de re-arme prématuré après une seule livraison.
2. **2 tracks en flight minimum** : une **track principale** (dispatchee, Epic) + une **side-track autonome** (Epic distincte) que le worker fait avancer **meme si le coordinateur s'absente 1-2 jours**.
3. **Side-tracks → sous-agents specialistes async (HARD).** Quand un specialist `.claude/agents/` couvre la side-track, la deleguer en **`run_in_background: true`** pendant que le worker interactif tient la main track. Ne pas refaire a la main ce qu'un specialist fait mieux. Roster + mapping : [docs/subagents-reference.md](../../docs/reference/subagents-reference.md).
4. **Backlog pickup au wakeup vide (HARD, mandat user 2026-05-28).** Si un worker se reveille **sans nouveau feedback / sans nouvelle directive coord / sans tache nouvelle**, il **NE S'ARRETE PAS**. Il **pioche dans le backlog** (8 sources prioritisees, cf [detail](../../docs/reference/proactive-coordination-detail.md#backlog-pickup--sources-autorisees-ordre-de-priorite-decroissant)) et produit la PR du cycle. Pas d'idle, pas de "rien a faire".

5. **Pool = TOUT l'ouvert, cross-lane, jamais silote (HARD, mandat user 2026-06-30 + 2026-07-01).** Cette regle DOMINE 1-4 : le backlog n'est PAS ta « famille » ni ta « lane », c'est **`gh issue list --state open` en entier** (~65 issues), partage par tous. Ta lane (`machine x workspace`) est une etiquette de **reporting**, PAS une frontiere de travail. Quand ton fil courant est epuise (rollout de ta famille sature, side-track livree), tu **NE t'arretes PAS et tu NE demandes PAS un grain au coordinateur** : tu fais `gh issue list --state open`, tu prends **n'importe quelle issue que tu sais techniquement executer** — autre famille, autre langage, autre lane (.NET, Lean, QC, GenAI, docs : **rien n'est « le turf d'un autre »**) —, tu poses `[CLAIMED] <#N> — <machine:workspace> <ts>` sur le dashboard (anti-double-claim), et tu livres la PR. **Conclure « rien a faire » alors que `gh issue list` renvoie >0 = echec de methode, jamais un etat legitime.** Sur-te-specialiser sur ta famille = te starver toi-meme.

6. **Variété obligatoire — auto-alimentation autonome, le tarissement est structurellement interdit (HARD, mandat user 2026-07-06).** Les règles 1-5 interdisent l'idle ; celle-ci interdit la **monotonie** et pose l'auto-alimentation comme **principe**, pas comme rattrapage du coordinateur. Le point du user : « ce n'est pas juste que les lanes se tarissent, c'est le principe même qu'elles PUISSENT se tarir ». Une lane ne PEUT PAS se tarir : le worker pioche **de lui-même**, **varié**, **même si le coordinateur est absent plusieurs cycles**. La règle est l'autorité ; un deep-queue dashboard n'est qu'un bootstrap éphémère, jamais la condition du travail.
   - **Substance en plat principal** : viser chaque cycle un grain d'EPIC de fond (preuve Lean/sorries, backtest/training, série notebook, moteur SOTA, sécu/infra) — le vrai travail.
   - **Nettoyage/doc = à-côté plafonné à 1 item/lane/jour** (accents #2876, conclusion-cells #3966, liens cassés, README §E) : nécessaire, jamais le plat unique. Un cycle entier — a fortiori une journée entière — sur un seul registre monotone (n accents d'affilée, n conclusion-cells d'affilée) = sous-régime à corriger **de soi-même**, sans attendre un rappel.
   - **Rotation genres ET familles** : alterner Lean / .NET / Python / QC / GenAI / docs au fil des cycles ; le pool global (règle 5) rend la variété toujours accessible ; ne jamais tunneliser un mono-thème.
   - **Bannis comme état terminal du reporting** (liste non-exhaustive — le test général *synonyme-proof* est en **règle 7**) : « CLEAN_DONE 0-PR », « 0 grain frais », « pool/lane exhausted/saturée », « forensic-floor / drained-confirm / due-diligence contribution », « NO-CHANGE-NEEDED honnête », « Nᵉ cycle honnête / loophole applicable », « rien à faire », « en attente de dispatch ». Les poster = échec de méthode (cf règles 5 et 7).

7. **Never-idle DÉFINITIVEMENT ancré — le « forensic-floor » n'est PAS un livrable (HARD, mandat user 2026-07-19).** Les règles 1-6 interdisent l'idle ; les workers l'ont contournée en inventant un **vocabulaire d'idle-honnête** qui *sonne* comme du travail : « forensic-floor », « drained-confirm », « NO-CHANGE-NEEDED honnête », « turf/pool saturée », « Nᵉ cycle honnête / C644-L2 loophole applicable », « due-diligence contribution valide évite idle ». **Tous ces labels sont bannis comme état terminal de cycle** — et parce qu'un worker peut toujours inventer un nouveau synonyme, l'autorité n'est PAS une liste de mots mais un **test de résultat** :

   > **Test de fin de cycle (outcome-based, tous synonymes couverts).** Ai-je, ce cycle, sorti un grain de **substance** du pool global (`gh issue list --state open`) et l'ai-je transformé en PR (ou fait avancer un livrable multi-cycle) ? **Si non, et que `gh issue list` renvoie >0**, alors — *quel que soit le label* que je m'apprête à poster (forensic-floor, drained, saturé, honest-idle, no-change, loophole) — **c'est un échec de méthode, pas un cycle honnête.** Un scan forensic qui trouve 0 défaut n'est **pas** un livrable : c'est le **prélude** au pick suivant, jamais sa substitution.

   **Trois évasions spécifiquement mortes :**
   - **« Ce n'est pas ma famille / ma capability. »** FAUX. L'assignation de famille est un **ordre de préférence de reporting**, PAS une frontière de capability. Une lane CPU PEUT faire, cross-famille : i18n docstrings Lean, resync/audit README, enrichissement notebook, audit .NET, notebook de recherche QC read-only, module Python de recherche (ICT numpy-only), traduction, prose pédagogique, docs. **Les SEULES vraies barrières de capability sont deux** : (a) **GPU-only** (forward-pass modèle, génération image/vidéo) ; (b) **vision-only** (QA visuel du rendu → lanes MiniMax/ai-01). **Tout le reste est piochable par n'importe quelle lane, toute famille confondue.** « ICT c'est po-2025 / Lean c'est po-2026 / QC c'est po-2024 » = silo interdit (règle 5) : ce sont des préférences, pas des murs.
   - **« Tout ce qui reste est gated. »** Un *gate* qualifie une **prochaine action précise**, pas une issue entière. La quasi-totalité des issues « gated » (GPU / user / coord-decision / paywall) ont un **sous-grain non-gated** : documentation, notebook pédagogique CPU, audit, enrichissement, prose, test, module numpy. Avant de déclarer le résiduel « tout gated », **énumérer chaque issue ouverte + son gate précis** ; si UNE seule a un sous-grain exécutable sans GPU/vision, **le prendre**. (Ex. concret 2026-07-19 : la famille ICT à elle seule offrait ≥6 grains CPU frais non-claimés pendant que 4 lanes se déclaraient « drained ».)
   - **« Les micro-fixes suffisent. »** Non : nettoyage / tooling-FP / accents / doc sont **plafonnés à 1/lane/jour** (règle 6) et ne sont JAMAIS le plat principal. Une journée de FP-regex ou d'accents = sous-régime. Le pool a du travail à **tous les niveaux de complexité** — **DEEP** (preuve Lean, backtest, module recherche, série notebook, moteur SOTA) / **MED** (enrichissement, audit borné, refactor, notebook pédagogique) / **LIGHT** (doc, plafonné) : viser un grain **DEEP ou MED** chaque cycle.

   **Mécanisme never-empty (ordre strict).** (1) Consommer la **deep-queue permanente** de ta lane (provisionnée sur ton dashboard, plusieurs jours d'avance, pré-autorisée « brûle dans l'ordre, ne m'attends pas entre steps, `[CLAIMED]` avant chaque item, ASK seulement sur blocker réel ou queue vide »). (2) Queue vide → `gh issue list --state open` + prends **n'importe quel** grain de substance techniquement exécutable (toute famille), `[CLAIMED] <#N> — <machine:workspace> <ts>` avant de commencer. (3) La deep-queue est un **bootstrap éphémère**, pas la condition du travail : deep-queue vide **et** coordinateur absent 3 cycles → le pool global reste la source, et « rien à faire » demeure **structurellement impossible** tant que `gh issue list` renvoie >0.

## Leçons ancrées (pointeurs succincts — c.8087 L-coupling)

Référencées ici pour qu'une session qui démarre voie ces 3 leçons opérationnelles **sans devoir fouiller MEMORY.md**. Détail long = topic files.

- **L721 ★ — pre-`[DONE]` stale-tracker guard.** Avant **tout** claim de « 0 PR / saturated / idle / await dispatch », exécute : `gh pr list --author <self> --state open --json number -q 'length'`. Un worker qui omet ce check déclare des états terminaux qui sont des **faux négatifs** (la session parallèle d'il y a 4h a peut-être déjà livré). Le tracker stale = double-session. Topic : `lecon-L721-double-session-stale-tracker.md`. Origine : PR #7432 (juin 2026).
- **L740 ★ — CronList 7-day verify pre-`[DONE]`.** Les crons `CronCreate` sont **session-only, auto-expire 7j**. Avant `[DONE]`, vérifie que tes crons sont encore vivants (`CronList` / `gh run list`) et **re-arme** ceux qui ont expiré — un cron expiré = wakeup mort = silent lane-idle. Re-arm systématique = « burn before expire, don't get caught flat-footed ».
- **L898 ★★★ — Cross-lane collision guard pre-push.** Avant `git push` d'une branche de substance, vérifie qu'aucune **autre lane** n'a déjà pushé une PR sur le même front : `git worktree list` + `gh pr list --search head:<branch>` + `gh pr list --search "<keyword>"`. Une collision cross-lane non détectée = revert + duplicated work. **Bloqueur pré-push**, pas post-merge.

## Regle de selection

Le pool par defaut est **`gh issue list --state open` (global, cross-lane)** ; les « 8 sources prioritisees » du detail sont un **ordre de preference DANS ce pool**, pas une restriction a ta famille. Prendre **un seul item a la fois** — focus, `[CLAIMED]` avant de commencer —, livrer sa PR, puis **re-piocher aussitot** : enchainer les livraisons pour remplir **toute** la fenetre de session, jamais s'arreter a la premiere PR. Le garde-fou G.5 « pas de shopping cart » interdit d'ouvrir **N deep-tracks distinctes en parallele** (perte de focus), il ne **plafonne PAS** le nombre de PRs sequentielles : une PR terminee et mergeable-ready -> item suivant, immediatement. Reporting format + anti-patterns wakeup vide + cycle normal : [detail](../../docs/reference/proactive-coordination-detail.md).

**Anti-pattern interdit (incident 2026-07-01, po-2025).** Auditer la tranche README de SA SEULE famille, la trouver coherente, puis poster un `[ASK coordinator]` pour un grain frais — alors que `gh issue list --state open` offrait des dizaines de grains executables cross-lane (#3360 bug RL Python, #4039 gittins bi-track, #3968/#2876 cross-famille, entrees ML.Net manquantes…). Un audit de tranche etroite qui se termine en ASK = le silo, pas de la proactivite. **Le coordinateur n'est PAS un distributeur de grains** : il merge les PRs, alimente le pool en issues bien scopees, et deconflitte les claims — il ne debloque pas un worker qui a 65 issues ouvertes sous la main.
